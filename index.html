  • Resumen del protocolo

    CANopen es un protocolo de alto nivel internacional y estandarizado por “CAN in Automation ( CiA )” en “EN 50325-4” (CiA301). Es creado para abarcar la capa 7 del modelo OSI referente a la aplicación.

    En este sentido, es una implementación en software que establece ciertos puntos clave:

    • Especifica la creación de una base para el intercambio de datos y comandos entre nodos de un CAN bus por medio de “Communication Objects (COB)”
    • Implementa mecanismos para intercambios de datos del proceso en tiempo real, transferencia de largos volúmenes de datos y envío de tramas de emergencia personalizadas.
    • Definir las interfaces de cada Nodo o Dispositivo para poder acceder a parámetros, ya sea en modo de lectura o escritura.

    Al estar enfocado en una capa más alta del modelo OSI, no interfiere de ninguna manera con el funcionamiento de los controladores CAN genéricos comerciales basados en ISO-11898-1 e ISO-11898-2, al contario, los toma como base de funcionamiento y trabaja directamente con estos estándares.

    Su uso está altamente aceptado para aplicaciones críticas como en el área industrial, aeronáutica y en dispositivos médicos como Máquinas de hemodiálisis.

    CANopen es mantenido y actualizado por una organización llamada CiA, que define la documentación y guías de uso del protocolo para las distintas aplicaciones.

    Con todo esto el protocolo se concibe como una alternativa que garantiza interoperabilidad y facilita el desarrollo de soluciones robustas y confiables

  • Ventajas
    • Al ser un estándar, es fácil encontrar guías y directrices para implementar cada una de las características del protocolo.
    • Resuelve la necesidad de crear y validar mecanismos complejos para interactuar eficientemente con los dispositivos en la red.
    • Al tener una estructura de mensajes e interfaces estandarizadas, es posible integrar interfaces de conexión para dispositivos de monitoreo y diagnóstico CANopen comerciales.
    • Define como integrar “data loggers” para almacenar e interpretar mensajes históricos importantes (Errores, valores de sensado, acciones) con timestamps para tener un auditorías más eficientes.
  • Desventajas - Retos
    • La curva de aprendizaje puede llegar a ser muy compleja
    • Implementar CANopen en un dispositivo, requiere la creación o integración de librerías que pueden no existir para marcas o modelos específicos de MCUs
  • Características técnicas - Trama y COB-ID

    El estándar proporciona la implementación de sus características al nivel de la capa de sesión con Cia 303-1, Presentación con CiA 303-2, y a nivel de Aplicación con CiA 401, 402, etc..

    En las primeras capas (Capa de enlace de datos y capa física), se utiliza el estándar ISO de CAN básico, siendo el ISO-11898-2 e ISO-11898-1

    En CANopen se utiliza la trama original del protocolo CAN básico, pero se le da una perspectiva diferente a los campos recibidos

    COB-ID: Este es el campo habitual de Message ID, sin embargo se hace una distinción de los primeros 4 bits con los siguientes 7, ya que aquí se agrega un concepto llamado “function code” para los primeros 4 y para los 7 bits siguientes es el “Node ID”

  • Seis conceptos núcleo de CAN Open
    1. Modelos de comunicación: Master-Slave, Client-Server, Producer-Customer
    1. Protocolos de comunicación: SDOs y PDOs (configuración de nodos y transmisión en tiempo real de datos)
    1. Estados de dispositivos: Estados que pueden cambiar en función de lo que requiera el nodo Maestro, ejemplo: Reset, inicialización, operacional
    1. Diccionario de objetos: Cada objeto tiene su diccionario de objetos con entradas específicas, por ejemplo uno para configuración accesible vía SDOs
    1. Hoja de datos electrónica: Un archivo con formato para los OD (Object Dictionary) permitiendo utilizar herramientas de servicio para actualizar dispositivos o monitoreo de la red
    1. Perfiles estándar de dispositivos: Módulos independientes de distintos fabricantes que generalmente son motion-control o actuadores con I/O

  • Modelos de comunicación

    Se proponen tres modelos clave para estructurar y facilitar la implementación en CANopen

    • Master-Slave: Un nodo maestro o host, se encarga de hacer envíos o peticiones a esclavos
    • Client-Server: Un cliente envía peticiones al servidor, el cual responde con los datos solicitados, por ejemplo cuando la aplicación maestra necesita información del OD de un esclavo llamandose una “subida” o una “descarga” según esta perspectiva
    • Consumer-Producer: Aquí el producer hace un broadcast en la red, el consumidor utiliza esta información y puede también ahora envíar sus datos como una petición Pull o siendo solo un Push sin petición
  • Objetos y perfiles

    La comunicación con CAN Open está basada en el uso de objetos para cada proceso, estos se definen en función de la tarea a realizar:

    • Identificación: Objetos con información del dispositivo como nombre o fabricante
    • Error: Objetos que contienen condiciones de error en el nodo
    • Comunicación: Objetos para transportar datos entre nodos
    • Configuración de Red: Objetos para asignar datos de configuración a los nodos
    • Gestión de Red: Objetos para monitoreo y control

    En un nivel más arriba, se definen perfiles de dispositivos estandarizados, lo cual vienen pre establecidos por el fabricante según una guía estándar

  • Protocolos o servicios

    Los protocolos o también llamados servicios enlistados a continuación, son los definidos para utilizarse en CANopen, en relación con su funcionamiento se le atribuye un modelo de relación (maestro-esclavo, cliente-servidor, productor-consumidor) para explicarlos individualmente.

    • Network Management (NMT)

      Este servicio es utilizado para controlar el estado de los dispositivos CANopen, permite dar órdenes de cambio de estado hacia otros dispositivos, los estados predefinidos por el protocolo son: Inicialización, Pre-operacional, operacional, paro… entre otros.

      Nota: Utiliza una relación Maestro-Esclavo

      En un ejemplo se envía un mensaje de 2 bytes con function code = 0, y un NODE ID = 0

      En este caso, un NODE ID = 0 indica un comando en Broadcast → COB-ID = 0

      Posteriormente la estructura de la trama define comandos predefinidos para indicar a N nodo que operación debe realizar, siendo estos por ejemplo, un comando de inicio, reset, stop.

      ANEXO:

      Adicionalmente, se pueden utilizar este tipo de mensajes como funciones de monitoreo al establecer el concepto de “Guarding” en donde el maestro solicita cíclicamente la respuesta de NMT status a los nodos para verificar que sea el estado correcto y que no pase un timeout sin recibir actualización. Aunque también los esclavos monitorean esta solicitud de su maestro, en caso de no recibirla pueden alarmarse y mandar frames de error al bus, además de realizar las acciones necesarias.

    • Synchronization (SYNC)

      El mecanismo de sincronización, puede implementarse utilizando el SYNC Frame, este es un mensaje enviado con un NODE-ID = 0, es decir, en broadcast a todos los nodos. Este mensaje se envía periódicamente en un tiempo definido por el nodo master.

      Al recibir el mensaje, los slave nodes suscritos, utilizan su diccionario de objetos predefinido para el envío de TPDOs (Transmit PDO), mensaje que contiene 8 bytes de información que puede contener variables de monitoreo importantes para el maestro.

      Es posible definir un contador en el SYNC Frame, que es un número de 1 byte y puede servir para seccionar la sincronización entre grupos de nodos.

      El COB-ID de este servicio es 80h

      El master node (Generalmente), se encarga de envíar este frame ciclicamente en intervalos definidos en el “communication cycle”

      https://www.can-cia.org/can-knowledge/canopen/special-function-protocols/

      CANopen offers three specific protocols for generating a special network behavior:

      • The SYNC protocol enables synchronous network behavior.
      • The Time-stamp protocol is used for the adjustment of a unique network-time.
      • The Emergency protocol can be used to inform other network participants about device-internal errors.

    • Emergency (EMCY)

      Las tramas de Emergencia utilizan el modelo producer/consumer se puede enviar desde un nodo que esté experimentando una situación adversa, ya sea error en alguna lectura, fallas de energía o cualquier improvisto. En ésta trama se añade información del error mediante un formato definido por el Object Dictionary

      Los COB-ID en este caso son: 81h - FF comenzando por el nodo 1

    • Timestamp (Time) (PDO)

      El frame de Timestamp, siendo un tipo diferente de PDO, incluye información de tiempo definida por el maestro. El formato incluye 4 bytes para representar milisegundos, los siguientes 6 bytes se utilizan para representar formato de fecha.

      Este frame puede ser utilizado para implementar un método de sincronización entre nodos esclavos y maestro, siendo que la tener el mismo “reloj”, son capaces de sincronizarse de una manera eficiente.

    • Process Data Object (PDO) - Communication Objects

      Dentro de el modelo de dispositivos se encuentra la definición de un diccionario de objetos para cada nodo particular, aquí se definen la estructura de una lista de parámetros para comunicación por medio de servicios como NTM, PDOs O SDOs

      El concepto de transmisión de datos en tiempo real “real-time data transfer” es posible a través del uso de PDOs. Este método está optimizado al no incluir parámetros de protocolo en los 8 bytes del mensaje, al contrario, se deja este campo abierto para transmitir toda la carga del mensaje útil y su interpretación o segmentación queda definida por el “mapping” que se haya realizado para el numero los PDO del nodo.

      “These include the process data object ( PDO) protocol used for the transmission of time critical process data”

      Se definen a gran escala dos tipos de PDO, los de Recepción (RPDO) y los de Transmisión (TPDO),

      Nota: Los PDOs son objetos de comunicación cíclica, es decir que suelen ser enviados durante los eventos de sincronización y en el diccionario de objetos se pueden definir un total de 32 TPDO con 32 mapping y 32 RPDO con 32 mapping

    • Service Data Object (SDO) - Communication Objects

      Este servicio utiliza la relación cliente - servidor, los clientes pueden solicitar datos a los servidores con los términos de petición y respuesta. Al enviar un SDO hay un protocolo distinto ya que aquí se indica el index y sub-index del Objeto al que se quiere acceder, dicho objeto debe estar definido en el Object Dictionary.

      Es parte del protocolo de este servicio definir si esta petición es para solicitar una descarga o carga de parámetros, al ser una descarga se le indica index y sub-index, a lo cual se debe envíar una respuesta

      Este servicio se utiliza en un modelo de cliente servidor, el cliente solicita al servidor una descarga de información, o una subida de información, esto implica el envío de peticiones y recepción de respuestas. Esto lo realiza en términos de interactuar con el Diccionario de objetos de cada dispositivo.

      Es utilizado para configurar parámetros de dispositivos, transferir bloques de información como actualizaciones, tener una comunicación confiable punto a punto P2P.

      En el siguiente ejemplo, el cliente indica el inicio de una descarga del nodo 5 por medio de enviar un R-SDO como broadcast a través de la red. Una vez recibido el mensaje por el nodo 5, este responde vía un Transmit SDO con el index del OD correspondiente. En caso de ser una carga, el nodo 5 respondería con la información correspondiente al index del OD Solicitado

    • Heartbeat (SDO)

      La función Heartbeat permite que el nodo maestro y esclavos envíen cíclicamente su estado NMT, esto es por iniciativa propia a través de un parámetro de tiempo modificable por medio de SDOs. Esto es la alternativa más utilizada en comparación con el modo “Guarding”, ya que el envío de los heartbeats son independientes entre nodos.

  • Funciones de monitoreo

    En CANopen se ofrecen servicios de monitoreo de la red para la detección de fallos o interrupciones de nodos. Se definen dos formas:

    • Guarding: Se utiliza el NTM para solicitar estatus periódicamente de los nodos
    • Heartbeat: Se refiere a la función brindada por el protocolo del mismo nombre

    El diccionario de objetos es un listado de estructuras de datos que se utilizan como interfaz para realizar la comunicación entre dispositivos con CANopen. Aquí se definen índices de 16 bits, en lo que se definen rangos reservados para los servicios o protocolos según el documento CiA DS 301, por ejemplo:

    La estructura general del diccionario posee el siguiente formato:

    • Índice: indica directamente el índice del objeto en el diccionario.
    • Objeto: describe que información el índice almacena (variables simples, array, record, etc.).
    • Nombre: contiene el nombre del objeto para facilitar su identificación.
    • Tipo: indica directamente el tipo de dato almacenado. Para variables simples, este tipo puede ser un entero, un float, etc. Para arrays, indica el tipo del dato contenido en el array. Para records, indica el formato del record, de acuerdo con los tipos descriptos en la primera parte del diccionario de objetos (índices 0001h – 0360h).
    • Acceso: informa si el objeto en cuestión está accesible solamente para lectura (ro), para lectura y escrita (rw), o es una constante (const).
